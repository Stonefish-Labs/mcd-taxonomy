# Source-to-Binary Drift

| Signal | Description | Effect |
|---|---|---|
| **Behavioral drift** | Compiled binary contains functionality (imports, strings, behaviors) not present in the published source | Very high signal — the binary was not built from the published source |
| **Build irreproducibility** | Binary cannot be reproduced from the published source and build instructions | May indicate post-build injection of malicious code |
| **Unexpected native extensions** | Package includes native binaries that are not explained by the package's functionality | Native code is opaque to source analysis and may contain hidden payloads |
| **Security mitigations disabled in compiled artifacts** | Compiled binaries shipped with exploit mitigations deliberately disabled — ASLR, DEP/NX, Control Flow Guard (CFG), stack canaries, RELRO, or SafeSEH turned off when the build toolchain enables them by default. Disabling these protections requires explicit compiler/linker flags (e.g., `-no-pie`, `/NXCOMPAT:NO`, `-fno-stack-protector`, `-z norelro`), indicating an intentional decision to weaken the binary's resilience to memory corruption exploits. Especially suspicious in native extensions distributed through package registries (`.so`, `.dll`, `.pyd`, `.node` files) where the consuming developer has no visibility into compilation flags. A native extension with ASLR and DEP disabled is substantially easier to exploit via buffer overflow or ROP chain than one with default protections. | High signal — deliberate weakening of exploit mitigations in distributed binaries has no legitimate justification in production code |
| **Unsigned or weakly-signed distributed binaries** | Compiled artifacts distributed without code signatures, or signed with deprecated algorithms (SHA-1, MD5) when stronger alternatives (SHA-256+) are available and standard. Unsigned binaries from a publisher that signs other artifacts in the same package indicate either a build pipeline inconsistency (worth investigating) or a deliberately unsigned payload. Deprecated signature algorithms may be vulnerable to collision attacks, allowing an attacker to substitute a malicious binary that produces the same signature hash. | Medium signal — absence of signing is common but noted; deprecated algorithms are medium-high when stronger signing is used elsewhere by the same publisher |
| **Debug information or symbols in production binaries** | Compiled artifacts shipped with debug symbols, verbose logging enabled, or development-only code paths compiled in. While not directly malicious, debug information aids reverse engineering and may leak internal paths, developer names, or build environment details. More importantly, binaries that produce debug output to the system debug log at runtime create an information leakage channel — any process monitoring the debug log can observe the application's internal state. | Low signal in isolation — common in poorly-configured build pipelines. Medium when combined with other drift signals |
