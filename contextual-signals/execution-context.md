# Execution Context Signals

| Signal | Description | Effect |
|---|---|---|
| **CI/CD environment targeting** | Code that specifically checks for CI/CD environment variables (`GITHUB_ACTIONS`, `CI`, `GITLAB_CI`, `JENKINS_URL`, etc.) before activating or modifying its behavior. CI/CD runners are uniquely high-value targets because they concentrate secrets: deployment keys, registry tokens, cloud credentials, and publishing tokens are all present in runner memory and environment variables. The LiteLLM payload gated its activation on `GITHUB_ACTIONS` — it only ran in the environment where secrets were richest. | Strongly elevates severity of any co-occurring POIs. Code that activates specifically in CI/CD is targeting the environment where credentials are most concentrated. |
| **Security tooling context** | Code executing within or distributed through security scanning tools (vulnerability scanners, SAST/DAST tools, IaC scanners). Security tools have *legitimate broad access* by design — they read environment variables, configuration files, and runner memory as part of their normal operation. A compromised security tool is the ultimate wolf in sheep's clothing. The Trivy compromise demonstrated this: the scanner's legitimate access became the attacker's credential harvesting platform. | Critically elevates severity. A compromised security tool has pre-authorized access to exactly the secrets an attacker wants. |
| **Privileged orchestration context** | Code executing with Kubernetes cluster-admin, Docker socket access, or cloud IAM roles with broad permissions. In these contexts, credential access and lateral movement POIs have dramatically higher blast radius — a single compromised container can become a cluster-wide compromise. | Multiplies blast radius of any detected behavioral pattern. |
